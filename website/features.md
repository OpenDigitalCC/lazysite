<section class="feat-cat"><h2>Authoring &amp; content</h2><p class="feat-blurb">Write pages as plain text and let lazysite turn them into a site.</p><div class="feat-grid"><div class="feat-card"><h3>Markdown, served instantly</h3><p>Drop a .md file in the docroot and it is served as a fully rendered HTML page on first request, then cached. No build step, no pipeline.</p></div><div class="feat-card"><h3>Template Toolkit variables</h3><p>Pages can carry dynamic values — site config, page front matter, computed expressions, conditionals and loops — resolved at render time.</p><a class="feat-how" href="/docs/features/authoring/tt-variables">How it works: TT variables &rarr;</a></div><div class="feat-card"><h3>Reusable content includes</h3><p>Inline shared Markdown partials, code files or remote fragments into a page so common content lives in one place.</p><a class="feat-how" href="/docs/features/authoring/includes">How it works: Includes &rarr;</a></div><div class="feat-card"><h3>Cached remote includes</h3><p>Includes can fetch remote content and cache it for a set time, so external data appears inline without a fetch on every request.</p><a class="feat-how" href="/docs/features/authoring/include-ttl">How it works: Include TTL &rarr;</a></div><div class="feat-card"><h3>Pull in remote pages</h3><p>A .url file makes lazysite fetch Markdown from a remote URL (such as a GitHub raw file) and render it through the same pipeline — so docs can live with the code.</p><a class="feat-how" href="/docs/features/authoring/remote-pages">How it works: Remote pages &rarr;</a></div><div class="feat-card"><h3>Rich layout blocks (fenced divs)</h3><p>Wrap content in :::named blocks that become styled containers — callouts, columns, hero sections — without leaving Markdown.</p><a class="feat-how" href="/docs/features/authoring/fenced-divs">How it works: Fenced divs &rarr;</a></div><div class="feat-card"><h3>Embed media (oEmbed)</h3><p>Paste a video or social URL and lazysite expands it into the proper embed automatically.</p><a class="feat-how" href="/docs/features/authoring/oembed">How it works: oEmbed &rarr;</a></div><div class="feat-card"><h3>List and scan pages</h3><p>Generate indexes, galleries and menus by scanning a folder of pages and looping over them — with optional filtering by tag, date or path.</p><a class="feat-how" href="/docs/features/authoring/page-scan">How it works: Page scan &rarr;</a></div><div class="feat-card"><h3>Site search</h3><p>A built-in search index and results page let visitors find content without an external service.</p><a class="feat-how" href="/docs/features/authoring/search">How it works: Search &rarr;</a></div><div class="feat-card"><h3>News feeds</h3><p>Publish RSS and Atom feeds generated from your pages, kept in sync as content changes.</p><a class="feat-how" href="/docs/features/authoring/feeds">How it works: Feeds &rarr;</a></div><div class="feat-card"><h3>Sitemaps &amp; registries</h3><p>Auto-maintained sitemap.xml and llms.txt registries so search engines and AI agents can discover every page.</p><a class="feat-how" href="/docs/features/authoring/registries">How it works: Registries &rarr;</a></div><div class="feat-card"><h3>Dynamic pages from query parameters</h3><p>Read URL query parameters in a page to vary its output — simple dynamic pages with no application code.</p><a class="feat-how" href="/docs/features/authoring/query-params">How it works: Query parameters &rarr;</a></div><div class="feat-card"><h3>JSON &amp; API output</h3><p>Serve a page as pure data with api mode, or as an unwrapped fragment with raw mode — the same content reused as an API.</p><a class="feat-how" href="/docs/features/authoring/api-mode">How it works: API mode &rarr;</a></div><div class="feat-card"><h3>Custom error pages</h3><p>Author your own 404 (and other status) pages as ordinary Markdown.</p><a class="feat-how" href="/docs/features/authoring/404-page">How it works: 404 page &rarr;</a></div></div></section>
<section class="feat-cat"><h2>Design: layouts &amp; themes</h2><p class="feat-blurb">Structure, style and content stay in separate layers that never touch.</p><div class="feat-grid"><div class="feat-card"><h3>Layouts and themes</h3><p>A layout.tt owns the HTML chrome; a theme on top supplies colours, fonts and assets; content stays in .md files. Swap any layer independently.</p><a class="feat-how" href="/docs/features/configuration/layouts">How it works: Layouts &rarr;</a></div><div class="feat-card"><h3>Design tokens</h3><p>A theme declares its palette and type as tokens in theme.json, auto-emitted as CSS custom properties — re-skin a site by editing values, not stylesheets.</p><a class="feat-how" href="/docs/features/configuration/theme-json">How it works: theme.json &rarr;</a></div><div class="feat-card"><h3>Per-page layout override</h3><p>Any single page can opt into a different layout from front matter — a landing page or print view without affecting the rest.</p><a class="feat-how" href="/docs/features/configuration/layout-override">How it works: Layout override &rarr;</a></div><div class="feat-card"><h3>Install &amp; publish themes</h3><p>Install layouts and themes from a catalogue, or publish your own — including pulling them from a remote source.</p><a class="feat-how" href="/docs/features/configuration/theme-publishing">How it works: Theme publishing &rarr;</a></div><div class="feat-card"><h3>Multiple views</h3><p>Define alternate renderings of the same content for different audiences or formats.</p><a class="feat-how" href="/docs/features/configuration/views">How it works: Views &rarr;</a></div></div></section>
<section class="feat-cat"><h2>Dynamic &amp; interactive</h2><p class="feat-blurb">Forms, accounts and payments are built in — no plugins to bolt on.</p><div class="feat-grid"><div class="feat-card"><h3>Forms with delivery</h3><p>Define a form in Markdown and bind it to a delivery target (email or a handler) — contact, signup and feedback forms without code.</p><a class="feat-how" href="/docs/features/authoring/forms">How it works: Forms &rarr;</a></div><div class="feat-card"><h3>Authentication &amp; access control</h3><p>Protect pages or whole sections behind sign-in, with the right state revealed per visitor.</p><a class="feat-how" href="/docs/features/configuration/auth">How it works: Auth &rarr;</a></div><div class="feat-card"><h3>Payments (x402)</h3><p>Gate content or downloads behind a payment using the x402 protocol, built into the core.</p><a class="feat-how" href="/docs/features/configuration/payment">How it works: Payment &rarr;</a></div><div class="feat-card"><h3>Navigation from a config file</h3><p>The site menu is a plain nav.conf file read by the layout — edit the menu without touching templates.</p><a class="feat-how" href="/docs/features/configuration/nav-conf">How it works: nav.conf &rarr;</a></div><div class="feat-card"><h3>Web manager UI</h3><p>A browser-based manager to edit content, themes, navigation and users without the command line.</p><a class="feat-how" href="/docs/features/configuration/manager">How it works: Manager &rarr;</a></div><div class="feat-card"><h3>WebDAV &amp; AI publishing</h3><p>Publish over WebDAV and a control API — the path an AI agent or script uses to manage a site with no GUI in the loop.</p><a class="feat-how" href="/docs/features/configuration/webdav">How it works: WebDAV &rarr;</a></div></div></section>
<section class="feat-cat"><h2>Operate &amp; maintain</h2><p class="feat-blurb">Files on disk, fast by default, and trivial to back up or move.</p><div class="feat-grid"><div class="feat-card"><h3>No database</h3><p>Files are the source of truth. Nothing to provision, dump or migrate — back up by copying the folder.</p></div><div class="feat-card"><h3>Fast by default</h3><p>Dynamic only on the first request; plain cached HTML is served after that, with fine-grained control over caching.</p><a class="feat-how" href="/docs/features/development/cache-management">How it works: Cache management &rarr;</a></div><div class="feat-card"><h3>Static export</h3><p>Run in generation mode to produce a fully static site you can host anywhere — the same content, no server process.</p><a class="feat-how" href="/docs/features/development/build-static">How it works: Build static &rarr;</a></div><div class="feat-card"><h3>Local dev server</h3><p>A built-in server renders any Markdown tree locally with live preview — no install needed to start.</p><a class="feat-how" href="/docs/features/development/dev-server">How it works: Dev server &rarr;</a></div><div class="feat-card"><h3>Link auditing</h3><p>Scan the site for broken internal links before you publish.</p><a class="feat-how" href="/docs/features/development/link-audit">How it works: Link audit &rarr;</a></div><div class="feat-card"><h3>Configuration in one file</h3><p>Site-wide settings live in a single lazysite.conf, with overrides where you need them.</p><a class="feat-how" href="/docs/features/configuration/lazysite-conf">How it works: lazysite.conf &rarr;</a></div><div class="feat-card"><h3>Version-control ready</h3><p>Everything is a file, so the whole site — content, layout and theme — lives in a git repository.</p></div></div></section>
<section class="feat-cat"><h2>Access, permissions &amp; governance</h2><p class="feat-blurb">Decide exactly who can do what, lock content down, and keep a record of every change.</p><div class="feat-grid"><div class="feat-card"><h3>Fine-grained capabilities</h3><p>Per-account grants control exactly what each user or token can do — edit content, manage themes or layouts, change config, create sub-users — checked once, server-side, across every surface.</p><a class="feat-how" href="/docs/auth">How it works: Auth &rarr;</a></div><div class="feat-card"><h3>Per-file access control</h3><p>Map any path to an owner with read and write lists (users or groups). Access only ever narrows, the owner is always allowed, and the same check binds the manager, WebDAV and the AI connector.</p></div><div class="feat-card"><h3>File locking</h3><p>Lock a file so no one else can alter it. One lock store is shared across the manager and WebDAV, so a save anywhere respects a live lock.</p></div><div class="feat-card"><h3>Account &amp; token expiry</h3><p>Accounts and machine tokens can carry an expiry, after which access fails closed — useful for time-boxed contributors and partners.</p></div><div class="feat-card"><h3>Sub-users &amp; delegation</h3><p>A partner can mint scoped sub-accounts; onward delegation requires holding the capability itself, so nobody can grant more authority than they have.</p></div><div class="feat-card"><h3>Audit trail</h3><p>An append-only log records who changed what, to what, when, from where, and the outcome — material changes only, never browsing.</p></div><div class="feat-card"><h3>Two-factor authentication</h3><p>Optional TOTP second factor with recovery codes, self-contained with no external dependency.</p><a class="feat-how" href="/docs/auth">How it works: Auth &rarr;</a></div><div class="feat-card"><h3>Protected internals</h3><p>Scripts, secrets and credential files can never be read or written through the content tools — a hard deny-list, enforced on reads as well as writes.</p></div></div></section>
<section class="feat-cat"><h2>AI &amp; automation</h2><p class="feat-blurb">Built for a human/AI partnership — agents publish through the same rules a person does.</p><div class="feat-grid"><div class="feat-card"><h3>AI publishing over WebDAV &amp; API</h3><p>An agent or script edits content over WebDAV and a JSON control API — no GUI in the loop — bound by exactly the same capabilities and ACLs as any human author.</p><a class="feat-how" href="/docs/features/configuration/webdav">How it works: WebDAV &rarr;</a></div><div class="feat-card"><h3>MCP AI connector</h3><p>A built-in connector speaks the Model Context Protocol, so assistants such as Claude can manage a site through first-class tools.</p><a class="feat-how" href="/docs/ai-connector-tools">How it works: AI connector &rarr;</a></div><div class="feat-card"><h3>OAuth for AI partners</h3><p>Web-based AI connectors authenticate via a minimal OAuth 2.1 server with PKCE — the person proves they may act as a partner, and no secret is ever typed into the third party.</p></div><div class="feat-card"><h3>One enforced core</h3><p>Every door — browser, API, WebDAV, MCP — translates to the same shared handlers, so a lock, an ACL or a capability holds identically whoever knocks.</p></div></div></section>
