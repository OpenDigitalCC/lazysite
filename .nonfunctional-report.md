## 3. Performance

**Method.** Reasoned against the CGI baseline; no profiler run.

**Findings.**

- `check_preview` does **no filesystem access on the common no-cookie
  path** â€” it returns before touching the secret file. With a cookie
  present: one secret read + one HMAC.
- Preview rendering adds no shared-cache cost: it reuses the protected
  path, so the page is never read from or written to the cache.
- `preview-grant`: one secret read, one HMAC, two `stat` calls
  (layout.tt, theme.json). Negligible.

**Recommendations.** None.

## 4. Security

**Method.** Trust-boundary review against the cookie/secret design; diff
secrets scan; `perlcritic --theme=security` (only the documented `:utf8`
items).

**Findings.**

- The preview cookie is HMAC-SHA256 signed with the per-install auth
  secret (the same primitive and file as the auth cookie), constant-time
  compared, and **fails closed** (no secret â†’ no preview). Forgery
  requires the secret.
- Minting is **manager-only**: `preview-grant` is a POST behind the
  manager auth and the existing CSRF gate. The user field written into
  the cookie is stripped to `[A-Za-z0-9_.@-]`, preventing `Set-Cookie`
  header injection.
- A preview render forces `Cache-Control: no-store` and neither reads
  nor writes the shared cache, so a preview cannot leak into what
  anonymous visitors or shared caches see.
- Cookie attributes: `HttpOnly`, `SameSite=Lax`, `Secure` under HTTPS,
  one-hour `Max-Age`. Layout/theme are validated to `[A-Za-z0-9_-]` and
  must exist on disk before a grant is issued.
- Blast radius of a leaked/forged preview cookie: the holder's own
  rendered theme for at most one hour. It cannot affect other visitors
  and grants no escalation.

**Recommendations.**

- Accepted residual: a preview cookie is not bound to the auth session
  lifetime, so it remains valid (â‰¤1h) after logout. Risk is cosmetic
  (theme view only); left as-is.
