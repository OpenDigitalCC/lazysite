# Non-functional report — lazysite (SM071 Phase 2: user & auth foundation)

Date: 2026-06-17
Scope: changed areas for SM071 Phase 2 (delegated sub-user accounts,
capability flags, token lifecycle). Sub-milestone close-out, not a
full-project audit, per `rules/nonfunctional-close.md`.
Auditor: Claude

Changed areas: `tools/lazysite-users.pl` (sub-user model, provenance,
delegation, disable/enable/cascade/reassign, capability flags, token
lifecycle, partner-create), `lazysite-auth.pl` and `lazysite-dav.pl`
(disabled + token-expiry enforcement). New tests `t/unit/users/07..11`.

## 1. Test coverage

**Method.** `prove -r t/` (982 green, +70 over the pre-phase 912). The
user tool and the enforcement points run as subprocesses, so
`Devel::Cover` instrumentation does not propagate (the limitation SM070
recorded); coverage is reported by branch.

**Findings.**

- `07-sub-users.t` - provenance, create_sub_users / delegate gates, error
  paths, API path.
- `08-account-management.t` - disable/enable, cascade, ancestry
  authorisation, reassign (incl. cycle + actor restriction), and the
  end-to-end disabled-denied-over-DAV path.
- `09-capabilities.t` - manage_themes/layouts/config set/get/independence.
- `10-token-lifecycle.t` - pairing → exchange → authenticate over DAV,
  single-use enforcement, expired-token 401, rotation invalidating the
  old token, expired-pairing-key rejection.
- `11-partner-create.t` - defaults, extras, the onboarding brief, that the
  minted pairing key exchanges, and the create_sub_users gate.

**Gaps / recommendations.**

- The manager Users-page UI for the new settings/actions is not built;
  there are no UI tests (the project tests the manager UI manually).
- A rotation grace window is not implemented (atomic replace); deferred.
- `Devel::Cover` numbers uncaptured for the same subprocess reason as
  SM070.

## 2. Code quality

**Method.** `perl -c` on the three changed scripts; `perlcritic
--severity 4` against the documented baseline.

**Findings.**

- All three compile clean.
- perlcritic flags only the codebase's accepted deviation classes
  (`RequireFinalReturn`, `RequireEncodingWithUTF8Layer`,
  `ProhibitExplicitReturnUndef`, `RequireArgUnpacking`,
  `RequireLocalizedPunctuationVars`, `RequireBriefOpen`,
  `ProhibitExplicitStdin`). No new policy class.
- Two real bugs were caught and fixed during the phase: a list-assignment
  gotcha that made `is_ancestor` always return 0, and a `token_expired`
  helper called in `lazysite-dav.pl` before being defined. Both now
  covered by tests.

**Recommendations.** None material.

## 3. Performance

**Method.** Reasoned against the CGI baseline.

**Findings.**

- Each account operation reads/writes the single `user-settings.json`;
  `descendants` / `is_ancestor` are O(accounts) over an in-memory hash.
- Pairing keys and access tokens hash at iterations=1 (256-bit secrets),
  matching the SM070 credential rationale.
- Enforcement adds one settings read per auth/DAV request (already read
  for the ui/webdav mechanisms).

**Recommendations.** None.

## 4. Security

**Method.** Trust-boundary review of the account model and enforcement;
diff secrets scan.

**Findings.**

- Provenance (`created_by` / `created_at`) is immutable; `managed_by`
  changes only via reassign. Delegation is split (`create_sub_users` vs
  `delegate_sub_user_creation`), so a sub-tree cannot grow creation
  rights it was never granted.
- Management ops authorise on ancestry via `managed_by` **when an actor is
  supplied**; CLI with no actor is the unrestricted operator.
- Disabled and token-expiry are enforced in both `lazysite-auth.pl`
  (login refused / cookie rejected) and `lazysite-dav.pl` (403 / 401),
  ahead of the mechanism gate. Both fail open on a missing/corrupt
  settings file, consistent with `ui_enabled`, to avoid operator lockout.
- Pairing keys are single-use and short-lived; access tokens expire and
  rotate; partners get a locked password (token-only auth). Secrets are
  sha256iter-hashed, shown once, never logged.

**Recommendations (ranked).**

1. **High - gate the account-management UI on actor injection.** The
   `action_users` proxy in `lazysite-manager-api.pl` forwards any action
   to `tools/lazysite-users.pl` with operator privilege and **no actor**.
   Exposing `account-disable` / `account-enable` / `account-reassign` /
   `account-create` in the manager UI as-is would let any manager manage
   accounts outside their own sub-tree, bypassing the ancestry model.
   Before those controls ship, `action_users` (or a dedicated path) must
   inject `actor => $auth_user` for the account-* actions. This is the
   gating item for the Users-page UI and is best resolved alongside the
   Phase 3 control-API auth model.
2. Add a rotation grace window if concurrent partners need it (currently
   atomic replace).

## 5. Documentation

**Findings.**

- `tools/lazysite-users.pl` usage covers every new command.
- `partner-create` emits a generated onboarding brief.
- **lazysite-layouts alignment review (W2):** no changes required in
  `/srv/projects/lazysite-layouts`. Phase 2 is account/auth work,
  unrelated to layout/theme authoring.

**Recommendations / open items.**

- **Manager Users-page UI** for the new capability flags, sub-user
  permissions, disable/enable, reassign, create-sub-user, and the
  onboarding download is not yet built. The capability-flag toggles are
  mechanical (they flow through the existing `settings-set` proxy); the
  account-management controls are gated on recommendation 4.1 above.
  Operators use the CLI meanwhile.
- User-facing auth docs (the sub-user model, capabilities, tokens) update
  with that UI; the full `theme-publishing.md` lands in Phase 3.
