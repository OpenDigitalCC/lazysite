# Non-functional report — lazysite (SM071 Phase 3: theme & layout management)

Date: 2026-06-17
Scope: changed areas for SM071 Phase 3 (per-object WebDAV authoring,
content-hash manifest, token control API, activate-with-backup, layout
specifics, rate limiting). Sub-milestone close-out per
`rules/nonfunctional-close.md`. This phase completes SM071.
Auditor: Claude

Changed areas: `lazysite-dav.pl` (per-object authorise, lzs:sha256
property, rate limit + Retry-After), `lazysite-manager-api.pl` (token
front-path, capability gating, actor injection, artifact-manifest/

`tools/lazysite-users.pl` (verify-credential), docs, SBOM, and the new

## 1. Test coverage

**Method.** `prove -r t/` (1035 green, +53 over the phase-2 982). The
endpoints run as subprocesses, so `Devel::Cover` is not propagated (the
SM070 limitation); coverage is by branch.

**Findings.**

- DAV: `08-layout-authz` (active/inactive per-object rule, capability
  gating, rest-of-lazysite denial, dav_scope orthogonality), `09-manifest`
  (lzs:sha256 = content hash, scoped to layouts), `10-rate-limit` (write
  throttle 429 + Retry-After, reads unthrottled, 423 Retry-After).
- Control API: `10-control-api` (token auth, capability gating,
  non-allowlisted refusal, CSRF exemption, cookie+token rejection, invalid
  token, actor-injection ancestry), `11-activate-backup` (validation, base
  409, lock 423, snapshot), `12-layout-activate` (compile gate, compatible
  pair), `13-rate-limit` (control-API 429 + Retry-After).
- Every new gate/branch is exercised, including the end-to-end mint →
  verify → render/authenticate paths from earlier phases.

**Gaps / recommendations.**

- The manager Users-page UI is still not built (no UI tests); now
  unblocked - see §4.
- Devel::Cover numbers uncaptured (subprocess), as before.

## 2. Code quality

**Method.** `perl -c` on all three scripts; `perlcritic --severity 4`
against the documented baseline.

**Findings.**

- All compile clean. perlcritic flags only the codebase's accepted
  deviation classes (`RequireFinalReturn`, `RequireEncodingWithUTF8Layer`,
  `ProhibitExplicitReturnUndef`, `RequireArgUnpacking`, `RequireBriefOpen`,
  `RequireLocalizedPunctuationVars`, `ProhibitNoWarnings`,
  `ProhibitExplicitStdin`). No new policy class.
- Manifest computation is shared (`_compute_manifest`); the activate
  machinery (`_set_theme_pointer`, `_snapshot_artifact`, `_prune_backups`)
  is reused by both theme- and layout-activate. Four bugs were caught and
  fixed during the phase (a missing `token_expired` helper in DAV, the
  user-tool path resolution, and two test-only errors).

**Recommendations.** None material.

## 3. Performance

**Method.** Reasoned against the CGI baseline.

**Findings.**

- The manifest is one SHA-256 pass per artifact file, on demand; the
  lzs:sha256 PROPFIND property is scoped to `lazysite/layouts/**` so
  content listings never hash.
- The rate-limit bucket is one small JSON read/modify/write under `flock`
  per write/token request; fails open on IO error.
- `layout.tt` is parsed (not rendered) only at activate/validate time.

**Recommendations.** None.

## 4. Security

**Method.** Trust-boundary review of the per-object rule, the token
front-path, and the actor injection; diff secrets scan.

**Findings.**

- Per-object DAV authorisation: active theme/layout read-only; only
  inactive artifacts writable; rest of `lazysite/` denied; `dav_scope`
  does not gate themes/layouts.
- Control API: token auth verified via the users tool (rejects disabled /
  expired); CSRF-exempt only for cookieless token requests; refuses
  cookie+token mixing; confined to the control-API action set; each action
  capability-gated; `config-set` allowlist excludes access-widening keys.
- Account management injects `actor=$auth_user` so a manager acts only
  within its own sub-tree (the Phase 2 finding, now resolved).
- Activate is validated, artifact-locked (423), base-manifest-conditional
  (409), and backed up before the flip.
- Per-token rate limiting; 429/423 carry Retry-After.

**Recommendations (ranked).**

1. **Build the manager Users-page UI (task #25) - now unblocked.** The
   actor-injection gate it depended on is implemented, so the capability /
   sub-user / disable / reassign / onboarding controls can ship safely.
2. `sbom.json` is **tracked** in the working tree although SM065 intended
   it release-generated (and `.gitignore` lists it). The strict check
   regenerates it; the committed copy drifts. Reconcile: `git rm --cached
   sbom.json` (and the same for `release-manifest.json` if applicable), or
   accept it as tracked and drop the gitignore line.
3. A token-rotation grace window (currently atomic replace).
