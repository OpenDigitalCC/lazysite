Release 0.4.40 - Per-page theme override; config + theme self-service; brief CLI

Implements four items from the theming experience report. Full suite green; perlcritic
clean; strict SBOM gate green.

## Per-page theme: override (SM120) - the headline

A page can now pin a theme in its front matter (theme:), preview-only and sanitised the
same way as layout:, falling back to the active/site theme; resolve_theme still gates on
layout compatibility so an incompatible pin renders as no theme rather than breaking.
This makes a theme explorer - or previewing a candidate theme on a single page before
activating it globally - a one-line change on the real layout, instead of bespoke
theme-bound demo layouts. The report's single highest-leverage suggestion.

## Config self-service for tokens (SM122)

A manage_config token can now read a safe subset of the site config (config-read:
layout, theme, webdav_enabled, site_name, manager_groups, ...) to self-diagnose instead
of inferring from HTTP codes, and config-set accepts an injection-checked safe subset
(webdav_enabled, layout, theme, nav_file) on top of site_name/url/search_default.

## Theme discovery (SM123)

Token clients may list installed themes and layouts (theme-list / themes-for-layout /
themes-list-all / layouts-available added to the token allowlist), and a new MCP
list_themes tool returns the themes installed across all layouts. Previously a connector
discovered installs by activating each in turn.

## Brief CLI + access planes (SM124)

lazysite-users.pl brief USERNAME prints the agent onboarding brief (a CLI method, as
asked), and the brief now states explicitly that a token's capabilities are independent
of any manager-group / operator status the account holds - the access-plane mismatch
that surprised both sides. (MKCOL already returns 405 for existing-allowed paths; 403 is
the deny-list for internal paths, which is correct.)
