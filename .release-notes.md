Release 0.4.53 - Manager CSS refreshes on upgrade (editor fix)

The root cause behind the "editor empty / page ends at extra" report. Full suite green;
perlcritic clean; strict SBOM gate green.

## The actual problem

A deployed site was serving an 11691-byte /manager/assets/manager.css while the release
ships 55844 bytes - a pre-SM109 stylesheet with none of the editor (or notification, or
plugin, or stats) CSS. The whole manager UI was running on a stale stylesheet; the editor
showed it worst, because its entire layout depends on .mg-editor-root from that file, so
it collapsed to nothing ("the page ends at extra").

Cause: manager.css is authored under lazysite/manager/assets/ - inside the Apache-denied
/lazysite/ tree - and was mirrored to the web-served /manager/assets/ by a copy step in
install.pl. That copy could leave a stale file behind on upgrade.

## The fix

manager.css (and any manager asset) is now shipped STRAIGHT to {DOCROOT}/manager/assets/
by the manifest, in the code bucket, so every upgrade overwrites it - no more drift. A
new classification rule handles it ahead of the general lazysite/manager rule; the
manager layout template still installs under lazysite/manager/ for the processor. The
install.pl copy step is removed, and any orphaned /lazysite/ copy is cleaned up.

After upgrading to 0.4.53, each site's manager.css refreshes to the current version, which
- together with 0.4.51's inlined editor layout as a fallback - restores the editor and
the rest of the manager's styling.
