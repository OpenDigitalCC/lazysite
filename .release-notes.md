Release 0.4.44 - Operator notifications (SM113)

A reusable notification system, with form submissions as the first producer. Full suite
green (1709 tests); perlcritic clean; strict SBOM gate green.

## Operator notifications (SM113)

The manager header now has a notification bell with an unread badge and a dropdown list,
so an operator sees when something needs attention without hunting for it.

- Store: an append-only logs/notices.jsonl (each notice carries ts, type, message,
  target, url) plus a per-operator last-seen marker (logs/notices-seen.json) for the
  unread count.
- API: `notices` returns the most recent 100 (newest first) with the operator's unread
  count; `notices-seen` stamps the marker. Both are operator-only (not exposed to token
  clients) and GET, so CSRF-exempt - the seen-marker is a harmless per-operator write.
- First producer: a successful form submission raises a `submission` notice (best-effort,
  never blocking delivery). The bell links it to the Plugins page.
- UI: polls on load, marks everything seen when the panel is opened, closes on an
  outside click.

## Future

The doc's wider vision - a documented plugin-facing API so any plugin can raise a notice
(failed build, expiring credential, audit anomaly), per-plugin opt-in, and SSE push
instead of polling (ties to SM103) - is noted for later cuts. This v1 ships the store,
the API, the badge, and the first producer.
