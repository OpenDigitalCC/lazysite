Release 0.4.55 - Deploy reloads nginx for fresh static assets

The final piece of the "blank editor" investigation. (Shell-only deploy change.)

## What was happening

With the ?v=<version> cache-buster from 0.4.54 live, a freshly-versioned asset URL on the
deployed site STILL returned the old byte count (manager.css 11691 not 55844,
codemirror.min.js 57640 not 170535). That proves it is nginx's PATH-keyed open_file_cache
- which ignores the query string - serving a stale cached file handle/size, even though
the files on disk are current. nginx was effectively sending a truncated stylesheet/JS,
so the editor's CodeMirror mounted invisibly.

## The fix

The Hestia deploy now reloads nginx after the permissions step. A reload spawns fresh
worker processes with an empty open_file_cache, so nginx re-reads the updated manager
assets. It falls back to reload-or-restart and, failing that, prints the manual command.

Together: the 0.4.54 cache-buster covers browser/CDN caches, and this covers nginx's
server-side cache - so manager assets refresh cleanly on every upgrade.

To clear the current stale cache on an already-upgraded site, restart nginx once:
systemctl restart nginx
