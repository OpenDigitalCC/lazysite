Release 0.5.38 - Reported-issue fixes: dev-server cookies + WebDAV denial reasons

Full suite green (2,089 tests, 145 files); perlcritic clean (severity 4 +
security theme); compile sweep green; bench gate green; coverage gate green;
strict SBOM gate green.  Channel: edge.

Two validated reported issues, both agent-facing:

- The development server dropped the second of two identical response headers.
  lazysite-auth.pl sends two Set-Cookie headers (the real session cookie plus
  a display marker), and the dev server's name-keyed parse kept only the last -
  so dev-server login never established a session and logout left one live.
  Production Apache/nginx were never affected. The server now forwards every
  header in the order the CGI emitted them, and its header parser is a testable
  modulino. (RI-001)

- WebDAV refusals now say why. A partner (typically a publishing agent) that
  was refused a write - a missing capability, a wrong path, or an attempt to
  edit the active theme/layout - got a bare "Forbidden" and had to guess. Each
  denial now names the cause in the body ("Forbidden: <reason>") and in a
  machine-parseable X-Lazysite-Deny-Reason header. Authorisation is unchanged;
  only the messaging improved. This directly addresses the reported
  theme-install trial-and-error. (RI-002)

The wider capability-discoverability asks behind RI-002 (a machine-parseable
capability map, theme/layout quickstarts, a private-file guardrail) are
recorded in the feature-request backlog for scoping.
