Release 0.4.51 - Editor survives a stale/missing manager.css

Resolves the persistent "editor empty, page ends at extra" report, diagnosed from the
deployed page source. Full suite green; perlcritic clean; strict SBOM gate green.

## Root cause

The editor's whole layout hangs off .mg-editor-root { position: fixed } in
/manager/assets/manager.css. That file is special: it is authored under
lazysite/manager/assets/ (the Apache-denied /lazysite/ tree) and copied to
{DOCROOT}/manager/assets/ by install.pl, rather than shipped there by the manifest. If
that copy is stale, missing, or unreadable - for instance after an interrupted
permissions sweep - .mg-editor-root loses its positioning, the editor renders inline,
and the content + preview panes collapse to zero height. The JavaScript runs fine (hence
no console error); the editor is simply laid out to nothing, so the page appears to end
at the last front-matter field ("extra").

## Fix

The edit page now inlines the critical editor layout - the fixed full-screen overlay,
the flexed editor/preview panes, and explicit min-heights for the content and preview -
in its own <style> block, which is part of the page HTML and therefore always applies.
It uses a hard-coded 48px header offset and a --mg-bg fallback so it does not depend on
manager.css's custom properties. The rules are higher-specificity (#id.class), so they
also win harmlessly when manager.css does load. The editor can no longer collapse just
because an external stylesheet failed to arrive.

If the wider manager also looked unstyled, that confirms /manager/assets/manager.css was
not loading; a clean redeploy on 0.4.48+ (with the permissions-sweep fix) refreshes the
copied stylesheet and its permissions.
