Release 0.4.41 - WebDAV route health check (SM121)

Addresses the recurring "WebDAV /dav/ returns nginx 404" report with a diagnostic.
Full suite green; perlcritic clean; strict SBOM gate green.

## The problem

The Apache vhost template already wires WebDAV (ScriptAlias /dav -> lazysite-dav.pl),
and lazysite-dav.pl does its own auth and honours webdav_enabled - so a correctly
provisioned site answers /dav/ with 401 even unauthenticated, never 404. A "404 even
unauthenticated" means the web server / nginx proxy is serving /dav/ as a static path
instead of forwarding it to Apache (a route/provisioning problem, not auth), or the
domain is nginx-only (lazysite needs the apache2 backend).

## The health check

lazysite-check gains --check-dav URL: an unauthenticated probe of URL/dav/ that reports

- OK   - 401 challenge: the route is wired.
- FAIL - 404: the web server / proxy is not forwarding /dav/ (the fastest signal that
         it is a route problem, not an auth/scope one).
- WARN - no response (curl missing or host unreachable).

Run it standalone or alongside the deploy:

    perl tools/lazysite-check.pl --docroot "$DOC" --check-dav "https://your-domain"

The Hestia runbook now documents the route requirement, the 404-vs-401 diagnostic, and
how to fix a 404 (apache2 backend + an nginx proxy that forwards /dav/). Auto-wiring the
proxy on enable is environment-specific and needs verification on the box; the health
check turns it into a one-line diagnosis.
