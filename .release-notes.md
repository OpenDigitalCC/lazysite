Release 0.4.20 - Deploy ownership + secret-perm hardening

Fixes the deploy and doctor issues surfaced rolling 0.4.19 across a Hestia fleet.
Full suite green; perlcritic clean; strict SBOM gate green; secrets clean.

Commits since v0.4.19 (oldest first):
- cd267b0  fix(SM093): deploy normalises ownership when root + protects secrets; doctor checks CGI-readable secrets
- a8d1a4d  fix(SM093): lazysite-check defaults the expected group to www-data

## Deploy: work the ownership out when run as root

lazysite-hestia-deploy.sh ran install.pl as the domain user, which failed with
"Permission denied" when the docroot/cgi-bin had been left owned by root (from an
earlier install.pl run directly as root) - the upgrade aborted half-done. The
deploy runs as root, so it now chowns the docroot + cgi-bin to <user>:www-data
BEFORE running install.pl, and the final verify runs lazysite-check --fix (as root)
to repair anything still off. A fleet update over a previously root-installed site
now upgrades cleanly instead of failing.

## Secrets are no longer world-readable

The deploy's blanket `chmod 664` exposed auth/.secret, forms/.secret, oauth.json
and user-settings.json to the world. They are now tightened to 660 (readable by the
www-data group only). lazysite-check already flags world-readable secrets; this
stops the deploy creating them that way.

## Doctor: catch secrets the CGI cannot read

lazysite-check now flags a secret that is not world-accessible but is also not
readable by the www-data CGI - e.g. 0600 owned by a non-www-data user, which is the
exact cause of an "End of script output before headers" 500 once a session cookie
is present (the wrapper reads .secret to verify the cookie and dies). --fix repairs
it. The expected group also defaults to www-data (not the docroot's group), so
--fix can never strip the CGI's group access - the regression that briefly bit a
manual chown.

## Rolling this out

Re-run your fleet update with this version. It will: chown each site to
<user>:www-data before install (fixing any site whose 0.4.19 upgrade failed on a
root-owned tree), set secret perms to 660, and verify+auto-repair. A site whose
0.4.19 upgrade failed (e.g. sovereigncomputing.org) upgrades cleanly on this pass.
