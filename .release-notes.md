Release 0.4.52 - Upgrades leave the Hestia web template untouched

A deployment-workflow fix from live use. (Shell-only change; suite unaffected. perlcritic
clean; strict SBOM gate green.)

## The problem

The per-site Hestia deploy re-ran v-change-web-domain-tpl ... lazysite-app yes on every
invocation. So lazysite-hestia-update-all - which deploys every site carrying the
lazysite install marker - re-asserted the lazysite-app template and forced a vhost
rebuild on each upgrade. That silently reverted any domain whose template had been
changed on purpose (for example, switched back to its original template to keep a static
SSI or PHP site working) and needlessly churned Hestia state during a routine code update.

## The fix

The deploy now applies the web template on FIRST-TIME setup only, detected by the absence
of the install marker (lazysite/.install-state.json). An upgrade leaves the domain's web
template assignment untouched and only refreshes code, content and permissions - so a
deliberately-changed template is preserved. To re-apply lazysite-app and rebuild the
vhost on purpose, run the deploy with LAZYSITE_APPLY_TEMPLATE=1.

update-all's --templates flag still refreshes the shared template FILES (so a vhost
change such as the SSI options can be staged), but it no longer rebuilds domains; rebuild
the ones you want with v-rebuild-web-domain USER DOMAIN (or the force env).
