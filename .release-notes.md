Release 0.5.33 - Fix: upgrade could overwrite an untracked homepage with boilerplate

Full suite green (1999 tests); perlcritic clean; strict SBOM gate green.  Channel: edge.

This fixes the incident where a homepage was replaced by the starter boilerplate
during an upgrade.

Cause: the installer preserves operator-edited content by comparing each file's
on-disk checksum against the checksum it recorded at the previous install. That
guard only ran for files it had previously recorded. A page that existed on disk
but was NOT in the install state - a homepage authored through the manager or
WebDAV, seeded by a different install path, or written before that file became
tracked - was treated as a brand-new file and the shipped boilerplate was copied
over it.

Fix: an untracked file that already exists on disk is now preserved (and adopted
into the state so it stays protected on future upgrades). The installer only writes
a file when the destination is absent, or when it is core code (which must match
the release). No content is overwritten.

You can preview exactly what an upgrade will do, without changing anything, by
running the installer with --dry-run: it prints how many files would be installed,
overwritten, or preserved, and lists the preserved ones.
