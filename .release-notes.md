## UTF-8 corruption - fully fixed

0.4.15 fixed send_json's :utf8 layer (the symptom shrank from double mojibake to
single), but a SECOND layer remained. A tools/call result puts $out in two places:
structuredContent => $out (already correct) and
content[0].text => encode_json($out). That inner encode_json emits UTF-8 bytes,
which the outer encode_json (in send_json) re-encoded - double-encoding non-ASCII
in the text part the client reads. The inner JSON is now decoded so the outer
layer encodes exactly once.

Also: the page-walk / search / preview / nav helpers read files as :utf8 (raw
reads would re-encode non-ASCII in snippets, previews, or rewritten links), and
STDIN is binmoded raw so decode_json owns the UTF-8 decode regardless of
environment.

Verified clean on the file bytes on disk AND the raw response bytes (read_file,
search_files, preview_page) - a write->read round-trip alone passed because the
write and read errors cancelled, so the regression test now checks the bytes.

After deploy, +/-, pound, accents, en-dashes and curly quotes round-trip through
the connector cleanly. /enquire can take a real +/- again (the ASCII workaround
is no longer required).

## read_nav / set_nav (completes the SM087 page API)

- read_nav - the navigation as a structured list (items + children) plus raw
  nav.conf.
- set_nav { items } - replace the navigation from an ordered { label, url } list
  (a children list becomes a sub-menu; an item with no url is a section header);
  written via action_save so it audits and rebuilds the cache.
