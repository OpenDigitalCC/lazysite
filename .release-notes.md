Release 0.4.26 - Fix AI-account Credentials; nav/forms capabilities; editable parent

Fixes a 0.4.25 regression and adds granular nav/forms capabilities plus an editable
account hierarchy. Full suite green; perlcritic clean; strict SBOM gate green.

## Fix: "Credentials: undefined" on AI accounts (deploy over 0.4.25)

0.4.25's Users-page refactor left the Credentials section's render outside its
human-only guard, so an AI/backend account showed "Credentials: undefined". Fixed -
the section renders only for human accounts. If you deployed 0.4.25, deploy 0.4.26.

## nav and forms as their own capabilities (SM105/SM106)

manage_nav gates nav editing; manage_forms gates form binding. Each inherits from
manage_content (which inherits the WebDAV grant) unless set explicitly, so nothing
changes for existing content editors - and you can now grant nav or forms on their
own (a navigation/chrome editor, or a forms owner, without full content write). Both
are toggles in the Users-page Publishing access group.

## Editable account hierarchy (SM104)

The Parent/Move control now shows for every account (not only existing sub-users), so
you can place a top-level account under another - this is how you move a user below
another. A sub-user's heading shows whose account it is under; top-level shows none.
The Add-user parent default was also reworded.
