Release 0.4.27 - Nav for token partners; feedback endpoint; do-not-retry

Unblocks navigation management for WebDAV/control-API partners, adds the agent
feedback endpoint and the do-not-retry signal, and fixes a doctor false-positive.
Full suite green; perlcritic clean; strict SBOM gate green.

## Navigation for token/WebDAV partners (the live 403)

A partner (e.g. Claude Code on the WebDAV/control-API path) could not manage the nav:
PUT /dav/lazysite/nav.conf returned 403 and read_nav was "not available to token
clients". Root cause was NOT the agent's "capabilities baked into the token" theory -
capabilities are read live per request (verify-credential -> effective_settings), so
a grant applies immediately. The real gaps, now fixed:

- nav-read / nav-save are token-client control-API actions, gated by manage_nav
  (which inherits manage_content -> webdav) - so any content-capable partner can edit
  the nav over the control API.
- the control-API whoami now reports the effective manage_content / manage_nav /
  manage_forms grants (previously omitted, so the partner couldn't see them).
- the agent brief has a "Managing the navigation" section (nav-read/nav-save, JSON
  body; not a WebDAV PUT to lazysite/, not MCP), and a stale "nav is editable over
  WebDAV with manage_config" line was removed.

The path for a partner: POST .../lazysite-manager-api.pl?action=nav-save with the
token as Basic auth and a JSON { items: [...] } body. No new pairing key needed.

## Agent feedback endpoint (SM102)

submit_feedback (MCP tool) writes an identity-stamped report to lazysite/feedback/ -
the agent supplies summary/good/bad/rating/context; the server stamps user, method,
ip, site, version and capabilities. Audited as 'feedback'. Reviewable when working on
lazysite (a manager Feedback page is the natural follow-on).

## Do-not-retry signal (SM101)

MCP tool errors now carry retryable: false for permanent failures (permission,
blocked, bad path, already-exists, ...) plus an imperative hint, so an agent stops
hammering instead of looping; only transient kinds (lock-held, rate-limited) are
retryable: true.

## Doctor false-positive fixed

The health summary flagged a manager/locks/*.lock as "foreign-owned". Runtime files
the www-data CGI creates (locks, cache, generated html, audit.log) are legitimately
www-data-owned; only a truly foreign owner (root, another user) is now a fault.
