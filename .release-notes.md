Release 0.4.38 - Form-build flow, client-side auth control, submission + denied audit

A cluster of features and fixes from the live review. Full suite green; perlcritic
clean; strict SBOM gate green.

## Agents can build forms natively (SM108)

The bind_form tool description now contains the whole flow - write front matter
"form: NAME" and a :::form block (field rules: required, email, textarea, select:A,B,
max:N, then a submit line), call list_form_handlers, then bind_form - with an inline
example, and the agent brief lists /docs/forms. So an agent builds a form from
scratch, not only by copying an existing page.

## Client-side sign-in/out (SM099)

The published site's auth control is resolved client-side from a non-HttpOnly
lzs_session marker cookie (display-only; the signed HttpOnly cookie is still the gate),
so a shared cached page never shows "Sign out" to an anonymous visitor (or vice
versa). The fallback layout uses it; any installed layout opts in with
data-ls-auth-in / data-ls-auth-out attributes.

## Submission + denied audit (SM115 + fix)

A form submission now writes a 'submit' audit event (origin form; the user is blank,
since the submitter is public). And a capability-denied attempt - an MCP tool or
control-API action the account lacks the capability for - now writes a 'fail' audit
event before refusing; previously it was silent, which is why an agent's blocked
theme-activity attempts left no trace. Successful theme/layout activation was already
audited on both paths. The concurrent-edit loss concern is already covered by the
editor's mtime conflict guard.

## Settings unsaved-changes reminder (SM118)

The Config settings form - the one surface needing an explicit Save - now flags
unsaved changes and warns on leaving while dirty, cleared on a successful save.
