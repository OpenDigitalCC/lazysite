Release 0.5.25 - Login page styling fix; audit is its own capability

Full suite green (1988 tests); perlcritic clean; strict SBOM gate green.  Channel: edge.

Login page styling
- The login page lost its form styling because inline CSS was being run through the
  Markdown emphasis pass: the `*/ ... /*` between two CSS comments was paired into
  `<em>` tags, turning the rules between them into one broken comment. The Markdown
  converter protected `<script>` blocks but not `<style>`; it now protects both, so
  CSS (and JS) is never Markdown-processed. This affects any page that carries a
  multi-line inline `<style>`, not just login.

Audit is a separate permission
- Viewing the audit trail was tied to the Analytics capability. It is now its own
  "Audit trail" capability, granted independently of visitor analytics (you can give
  someone the audit trail without visitor stats, or the reverse).
- New sites: the admins group gets it by default.
- Existing sites are left as-is on upgrade, so after deploying grant "Audit trail"
  to whoever should see it - on the Groups page, add the action to a group (your
  admins group is the obvious one) and put the user in it. Granting it needs the
  Users-and-groups permission, not audit itself, so no one is locked out.
