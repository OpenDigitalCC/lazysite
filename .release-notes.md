## Material audit trail

The audit was acting like an access log. It now records material events only -
state changes and security grants, not browsing - so it does not overlap the web
server access log (analytics belong in a future stats plugin, SM083).

- WebDAV reads (GET/PROPFIND) are no longer audited.
- The control API audits material POSTs only; user management is logged with the
  sub-action + target username (user-add, user-settings-set, ...), reads skipped.
- An OAuth token issue records a `connect` event - the "X connected" signal a
  read-only connector session (e.g. ChatGPT browsing) was missing.
- File writes read as the actual event: create / edit / delete / move / mkdir,
  across the control API, MCP tools and WebDAV.

## Connector robustness (from live ChatGPT testing)

- The task prompt now tells the assistant to confirm a write with read_file
  through the connector, NOT by fetching the rendered page - a separate slow
  request that could stall (the apparent "hang after the first edit"). The
  published page re-renders for visitors automatically.
- read_file refuses a file over 512 KB (a top-of-file cap) rather than returning
  a slow/oversized reply that could trip a client's per-call timeout. Binary
  files were already refused.
- New invalidate_cache MCP tool: the AI can force a re-render of a page (or "*"
  for all) - for pages that embed another, or to refresh a stale render.

## Manager UI polish

- The open file rights-editor is bracketed top and bottom by an accent rule, and
  its expander chevron turns accent-blue while open (easy to find the open file).
- In the ACCESS badge the @group indicator sits beside the owner.
- The Users page explains the two access domains sharing one account set: file
  management (internal - manager UI, control API, AI connectors) and site access
  (external - visitor/member login).
