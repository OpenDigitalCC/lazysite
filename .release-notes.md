Release 0.4.4 - Audit WebDAV reads; document MCP + per-client connection modes

A small visibility + onboarding release on top of 0.4.3 (the MCP server itself
already shipped in 0.4.3). Full suite 1481 green; perlcritic clean; strict SBOM
gate green (182 components); secrets gate clean. dist/lazysite-0.4.4.tar.gz
built (152 files); 0.4.3 tarball removed.

## Audit: WebDAV reads are now recorded

GET / PROPFIND over WebDAV are recorded in the shared audit trail (origin=dav),
so a partner's authenticated browse/read activity is visible - previously a
partner reading files left no trace. Default-on; set `audit_reads: false` in
lazysite.conf to quiet a busy site. (WebDAV/control-API auth is stateless/
per-request, so every audited line is itself an authenticated request - the
first read timestamp is when the partner started; there is no separate login
event.)

## Onboarding: MCP mode + per-client guidance

The partner briefing (ai-briefing-publishing) and the machine-readable bootstrap
(.well-known/ai-partner) now document both connection modes - API (WebDAV +
control API) and MCP (the lazysite-mcp.pl remote connector) - and which client
uses which: Claude.ai reaches a site only via a connector (MCP mode); Claude
Code / Desktop can use either; scripted clients use API mode. The mcp endpoint
is advertised in the bootstrap.

## Rollout

Same as 0.4.x - refresh the template + deploy every site:

    tar xzf lazysite-0.4.4.tar.gz -C /tmp
    /tmp/lazysite-0.4.4/installers/hestia/lazysite-hestia-update-all.sh \
        --templates /tmp/lazysite-0.4.4

After this, the sites' audit logs capture WebDAV reads, and the onboarding docs
describe MCP mode (the MCP endpoint itself has been live since 0.4.3, so you can
test the Claude.ai connector against an already-deployed site at
https://<site>/cgi-bin/lazysite-mcp.pl with bearer <partner-id>:<lzs_ token>).
